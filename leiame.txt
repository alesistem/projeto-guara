 RESUMO DO QUE FOI FEITO

O projeto foi recebido já com uma base de Chat funcionando

Detalhes de cada uma abaixo.


------------------------------------------------------------
1) AGENDA — GOOGLE CALENDAR
------------------------------------------------------------
O que faz:
  Cada usuário pode conectar sua própria conta Google e, pelo
  sistema, listar, criar e apagar eventos na agenda — incluindo
  convidar participantes por e-mail (eles recebem convite real
  do Google).

Arquivos novos:
  - googleAuth.js       → configura o OAuth2 da Agenda
  - googleCalendar.js   → funções de ler/criar/apagar eventos

Tabela nova no banco:
  - google_tokens        → guarda o token de acesso de cada usuário

Rotas novas:
  GET    /auth/google?usuario_id=:id
  GET    /auth/google/callback
  GET    /usuarios/:id/google/status
  GET    /usuarios/:id/eventos
  POST   /usuarios/:id/eventos
  DELETE /usuarios/:id/eventos/:eventoId


------------------------------------------------------------
2) LOGIN SOCIAL COM GOOGLE (JWT)
------------------------------------------------------------
O que faz:
  Qualquer e-mail Google autorizado (ver aviso no topo) pode
  fazer login no sistema. Se a pessoa nunca logou antes, uma
  conta é criada automaticamente (nome, e-mail e foto vêm do
  Google). Depois do login, o sistema entrega um token (JWT)
  que comprova quem é a pessoa.

Arquivos novos:
  - googleLogin.js      → OAuth2 dedicado ao login (escopo simples,
                           só pega nome/e-mail/foto, não toca a agenda)
  - authUsuarios.js      → cria/reconhece usuário, gera o JWT, e tem
                           o "porteiro" (middleware exigirLogin) que
                           protege rotas que exigem login

Colunas novas na tabela usuarios:
  - google_id   → identifica a conta Google da pessoa
  - papel       → 'aluno' ou 'professor' (definido manualmente
                  no banco por um admin, não tem tela pra isso)

Rotas novas:
  GET /auth/login/google
  GET /auth/login/google/callback
  GET /usuarios/me   (retorna os dados de quem está logado)

Observação técnica:
  A rota /usuarios/me precisa estar declarada ANTES da rota
  /usuarios/:id no server.js, senão o Express confunde "me"
  com um ID e dá erro. Isso já foi corrigido, mas vale lembrar
  caso novas rotas /usuarios/* sejam criadas no futuro.


------------------------------------------------------------
3) TURMAS (estilo Google Classroom)
------------------------------------------------------------
O que faz:
  Professores criam turmas e recebem um código de convite único
  (6 caracteres). Alunos usam esse código para entrar na turma.
  Ao criar a turma, um chat é criado automaticamente e vinculado
  a ela.

  IMPORTANTE: isso NÃO usa o Google Classroom de verdade — as
  turmas são só do nosso próprio banco. O Google é usado apenas
  para login/identidade, não para importar turmas reais.

Arquivo novo:
  - turmas.js   → criar turma, gerar código, entrar com código,
                  listar turmas e membros

Tabelas/colunas novas:
  - turma_membros          → liga usuário a turma + papel na turma
  - turmas.codigo_convite  → código único de 6 caracteres
  - turmas.criado_por      → quem (professor) criou a turma
  - turmas.chat_id         → chat vinculado automaticamente

Rotas novas:
  POST /turmas                    (só professor pode criar)
  POST /turmas/entrar             (aluno usa o código)
  GET  /turmas?usuario_id=:id     (lista turmas do usuário)
  GET  /turmas/:id/membros        (lista quem está na turma)


------------------------------------------------------------
4) CHAT EM TEMPO REAL (Socket.IO)
------------------------------------------------------------
O que faz:
  O chat já existia (mensagens via REST), mas não era em tempo
  real — precisava recarregar a página para ver mensagem nova.
  Agora, quem está com o chat aberto recebe a mensagem na hora.

  Como funciona: quando alguém manda mensagem, ela continua
  sendo salva normalmente no banco (POST /chats/:id/mensagens);
  depois disso, o servidor avisa via socket todo mundo que está
  naquele mesmo chat, sem precisar atualizar a página.

Arquivo novo:
  - socket.js   → configura o Socket.IO, controla quem entra/sai
                  de cada "sala" de chat, e avisa quando chega
                  mensagem nova

Arquivo de teste (não é parte do sistema final, é só pra testar):
  - teste_chat_socket.html   → página simples com duas caixinhas
                               de chat pra testar o tempo real

Mudanças no server.js:
  - servidor passou a usar http.createServer(app) em vez de só
    app.listen(), porque o Socket.IO precisa "andar junto" com
    o Express
  - a rota de enviar mensagem agora também avisa o socket depois
    de salvar no banco
